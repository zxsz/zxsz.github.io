web服务器基础

http简介

http全称为 Hyper text transfer protocol 超文本传输协议, 早期 http0.9 版本中只能传输文本信息, 传输的超文本html Hyper text mark lanuage ,而现在的web页面能显示各式各样的多媒体资源, 这是因为现在基本使用的时http1.1 版本, 服务器段能将多媒体资源(二进制格式) 转换成文本格式传输, 而客户端(浏览器) 能将种格式转换回媒体格式,从而显示在浏览器上这种机制称作MIME Multipurpose Internet Mail Extension.

浏览器是如何获取web服务器资源

互联网上存在大量的资源,如何使浏览器找到自己所有期望的资源,就需要一个同一的规范,于是URL就诞生了

URL 全称为 uniform resource locator

URL的语法

  • scheme 方案,例如 http , ftp ,smtp 等
  • user:password 用户名和密码,一般http都是匿名登陆所以很少看到,但有的ftp 服务器需要提供
  • host:port 表示服务器和端口,host可以使ip或者域名
  • path 表示主机下的路径
  • params 表示参数字段,某些协议可能需要传入一个参数,很少见
  • ?query 在一些动态网站下可能会提供站内搜索的功能,提供需要查询的条件
  • frag 用#引用表示 在一些特别长的网页下快速定位到某一个片段下

http的协议版本

  • HTTP/0.9

    http协议的最早版本,不支持mime类型,http的首部,只能通过GETt或和简单的html文档

  • HTTP/1.0

    支持了PUT POST DELETE GET ,并且引入了mime机制,从而支持了多媒体在通过http协议传输

    MIME 作用: 将非文本数据在传输前重新编码为文本格式，接受方能够用相反的方式重新还原为原来的格式。还能够调用相应的应用程序来打开此文件, 使用的编码使base64

  • HTTP/1.1

    目前使用的协议版本,主要是为了优化http措施,增加各种功能和提高效率

  • HTTP/2.0

    不久将会使用和推广的版本

http协议的一些请求方法

在实际中,你可能向浏览网站的某些内容. 或者登陆某些网站的会员, 或者上传博客,图片等这些动作都叫做http的请求方法, 它们被封装在http的首部中,以便服务器能够识别

常见的请求方法由以下几种:

method description
GET 最常用的请求,表示要浏览摸个网页资源
HEAD 于get类似不过head只是需要网页资源的首部信息,不需要主体部分
PUT 于get相反,向服务器写入文档,网页然后存储在服务器上
POST post于put类似不过只是提交一些用户信息等,例如通过html表单提交用户账号密码
DELETE 用于请求执行删除资源的操作
TRACE 主要用于诊断,验证请求经过的代理,网关等
OPTIONS 表示向服务器请求告知客户端支持哪些功能,或者获取资源的方法

状态返回码

访问一些网页的有时候会遇到 404 not found 和 502 bad gateway 的字样,这个就是状态码,在响应报文中体现的,

通过一些状态码可以快速定位一些问题的原因, 常见需要知道的状态码有

status description
200 ok 请求资源正确响应
301 Moved Permanently 请求的资源已经被永久重定向到别处,响应首部location中包含了新的url
302 Moved Temporarily 请求的资源已经被临时重定向到别处
304 Not Modified 请求的资源没有改变,表示浏览器中有此网页的缓存而且与服务器上的一致
401 Unauthorized web服务器中设置需要用户名认证的时候返回这个响应码
403 Forbidden 请求被拒绝, 客户端没有权限访问此资源
404 Not Found 请求资源
500 Internal Server 服务器内部错误
502 Bad Gateway 服务器作为代理或网关使用时,与下一个链路的服务器不通

与大多协议一样http协议也有自己的首部信息,http协议是工作在应用层上的,且首部是文本格式非常容易看懂.

首部headers的类型大致有以下几种 通用首部 请求首部 响应首部 实体首部 扩展首部

通用首部

通用首部即可以用在请求报文和响应报文中的,通常表示客户端和服务器之间连接的一些有用的信息

header description
Connection 指定客户端和服务器连接相关的选项,例如: keep-alive 长连接
Date 报文创建的时间
MIME-Version 发送端使用的mime版本
Transfer-Encoding 告知接受方使用的编码格式
Via 显示报文经过的中间代理或网关等
Cache-Control 控制缓存用的 例如 Cache-Control : public,max-age=600
Pragma 另一种跟缓存相关的,但是也有别的用处

请求首部

请求首部只在请求报文中, 发送了一些客户端的基本信息, 可以接受那些编码格式,语言,字符集,可接受那些压缩类型等.

header description
Client-IP 客户端ip地址
From 客户端用户的email地址
Host 请求的主机(fqdn) 如果单ip的服务器中有多个网站,就是靠这个识别
Referer 从那个超链接跳转过来的, 可以通过这个参数设置防止盗链
User-Agent 客户端程序一般指浏览器,
Accept 可接受的mime类型
Accept-Charset 可接受的字符集
Accept-Encoding 可接受的压缩类型 gzip deflate 等
Accept-Language 接送的语言
Expect 允许客户端列出某些请求所要求的服务器行为
If-Match 如果实体标记ETag与文档当前实体标记匹配就执行请求的动作
If-Modified-Since 后跟时间,如果请求的对象再这个时间之后修改了,就执行请求的动作
If-None-Match 如果资源修改过,就执行请求的动作
If-Range 允许对文档的范围进行条件请求
Range 如果服务器支持范围请求,就请求资源的指定范围
Authorization 客户端向服务器请求认证的数据
Cookie 客户端向服务器发送的cookie信息,主要让服务器记住客户端的身份
Max-Forward 将请求发送给服务器中间经过代理的最大次数
Proxy-Authorization 代理进行认证时使用的
Proxy-Connection 代理进行连接时使用的

响应首部

响应首部在响应报文中,主要用于告知客户端,响应的服务器一些基本信息,和一些特殊的功能,以便客户端能够发起更好更有效的请求报文

header description
Age 响应持续的时间
Public 服务器支持的请求方法列表
Retry-After 如果资源不可用在此时间或日期重试
Server 服务器应用程序的软件名称和版本
Title html文件的title
Accept-Ranges 服务器可接受的范围类型
Vary 服务器挑选最合适的资源版本发送给客户端
Proxy-Authorization 代理对客户端的质询列表
Set-Cookie 给客户端发个id号,以便服务器识别
WWW-authenticate 服务器对客户端的质询列表 401的返回码中有这个首部

实体首部

实体首部可以存在与请求报文和响应报文中, 通常是告知接受方如何处理报文中主体部分的内容

header description
Allow 列出可以对实体执行的请求方法
Location 前面状态码301,302 提到过, 这是服务器对资源重定向后,给出新的url地址
Content-Base 实体使用的url
Content-Encoding 实体的编码法师
Content-Language 实体最合适的语言类型
Content-Length 实体的长度
Content-MD5 实体的md5指纹
Content-Type 实体的对象类型
ETag 与实体相关的标记,例如: 请求的一个服务器页面html文件, 服务器会把这个页面的ETag发给客户端, 客户端将这个值跟当前缓存页面的ETag对比,就知道这个html文件有没有改变,没有改变就可以使用缓存的页面
Expire web服务器表明该实体将在什么时候过期
Last_Modified 缓存服务器中实体最后一次修改的时间

web服务器响应请求的过程有哪些

web服务器接受请求的步骤大致简单说可以分为以下几个步骤

  • 建立连接
  • 接受请求
  • 处理请求
  • 访问资源
  • 构建响应
  • 发送响应
  • 记录日志

第一步建立连接

客户端请求连接服务器首先是浏览器中输入某个网站的域名 例如: www.51cto.com 这是客户端首先是将域名转换成ip地址,可能是hosts文件,本地的缓存,dns服务器的缓存. 浏览器智能的将这个请求转换成标准的url格式添加默认80端口,和http协议 http://59.110.244.199:80, 让后构建一个请求报文, 这个请求报文再经过下成的通信子网(tcp层,ip层,数据链路成等) 与服务器建立连接, 当然具体是要先经过三次握手后才能建议连接.

其实tcp为了提高其传输效率客户端第三次向服务器握手的同时,可以允许将http协议的请求报文封装再tcp的数据段里了

第二步接受请求报文

报文到达服务器后会一层一层解封装, 服务器就会解析报文中的数据. 这些工作一般都是web服务器中的一个进程或线程去处理

需要注意的是,一些繁忙的web服务器上,可能同时有上万个请求同时到达,不对不同的web服务器有不同的接受请求的方式

  • 单线程的web服务器

    一个进程只处理一个请求,处理完成后才会轮到下一个

  • 多进程以及多线程的web服务器

    服务器预先生成若干进程每个进程再生成若干线程, 每个线程响应每一个请求

  • 复用的I/O 的服务器

    与上面的不同的是,上面的线程在响应请求的过程中处于阻塞状态, 线程程处于不可中断睡眠状态, 不能再去响应别的请求, 只能处理完这个请求后才能接受新的请求

    而复用的I/O,则表示一个线程响应请求后,直接扔给一个代理去处理,代理等待内核去磁盘上的文件后再由这个代理给线程, 而线程不需要傻等,可以继续响应别的请求. 提高了效率

第三步处理请求/访问资源

如上例子 请求的是网页的根目录, 浏览器构建的请求报文前面一行可能是 GET / HTTP/1.1 ,服务器就会知道客户端请求的动作是 GET 请求的资源是 / 客户端支持的协议版本是http1.1

解析完请求报文后, 服务器就会去访问对应目录下的资源, 对于这个报文请求的资源是 /

需要注意的是: 此处的 / 并不是服务器文件系统上的 / 而是web服务器程序中定义的 /

例如: httpd程序配置文件中的默认配置指令 DocumentRoot /var/www/html/ /var/www/html 就是访问站点的 /

nginx程序中指令Root /web/ /web/就是站点的 / ,这种方法叫做资源的映射

还要注意的是目录一般不是访问的资源, 在当客户端访问一个目录的时候会将访问目录下面的index.html 或正index.php文件响应给客户端, 例如: httpd 配置文件中的指令 DirectoryIndex index.html index.php

第四步构建响应

web服务器的程序获得资源后就会开始构建响应报文, 响应报文的格式前面已经提到 , 访问的资源作为响应报文的主体部分. 并加上http协议版本 响应状态码 原因短语 和一系列的首部信息

例如: Content-Type: text/html 表明实体的mime类型, Content-Length: 277 表明实体的长度 等等

第五步发送响应

web服务器程序构成响应报文后, 再经由下层的协议封包发给客户端, 之后的这写内容都是内核控制发送给客户端的, 需要注意的是: 发送完响应报文后,有的web服务器不是马上关闭tcp的连接状态, 现在大多数的web服务器都采用了keep-alive 的机制, 这是为了重用tcp的连接避免tcp的三次握手浪费时间,用于提高效率, http1.0版本后才引入的这种机制

记录日志

web服务器会将每个访问日志或错误日志记录再文件中, 这些信息一般都是重客户端的请求报文的首部,和响应报文首部中提取出来的例如: 记录客户端的ip, 客户端的浏览器类型, 访问的url, 时间, 从哪个站点的url跳转过来. 这些都是很有用的信息.
